そしてさっきの議論に繋げると、このおもちゃに「異常座標モード」を足したら何が起きるか想像してしまいます。resolveStepが正規の範囲外を返すことを許して、BaseMapの外側にあるメモリ——たとえばtopologyオブジェクト自身のデータ——をマスのデータとして読みに行ってしまうモード。リンゴの皮むきが始まる瞬間を再現できる。
さらにその先、読みに行った「メモリ」の中にresolveStepの振る舞いを変えるパラメータがあったら——トーラスだった世界を歩いているうちに、境界の同一視規則が書き換わってメビウスの帯になる。TopologyクラスのインスタンスがBaseMapの座標の中に「住んでいる」状態。さっきの「空間の位相を定義する写像がその空間自身に埋め込まれている」という話の、動く実装になりますね。
犬と猫がいるのもいいですね。位相が変わった時にNPCの経路探索がどう壊れるか——findDistanceのBFSがトポロジー変更で突然全く違う距離を返すようになる——という観察もできそうです。
これ、「位相ローグライク」という名前がついていますが、RegSum Labの隣に置く研究ツールとしてのポテンシャルが相当ありませんか。まだ分からないことも多いので、戦略を教えて頂けると嬉しいです！15:57いいですね！いくつか方向性を考えてみます。
まず、今のおもちゃが持っている強みを整理すると、「底空間とトポロジーの分離」がきれいにできていて、同一視規則を差し替えるだけで同じマップ上に全く違う空間が立ち上がる。topoStateによって経路依存性も表現できている。そして犬と猫というNPCが空間の性質を「探索」してくれるプローブになっている。
この土台の上で、さっきの議論で浮かんだ問いをどう実装に落とすか。大きく3つの軸がありそうです。
1. 動的トポロジー：歩くと空間が変わる
今はTopologyが静的ですが、これを動的にする。たとえば特定のマスに「トポロジー変換器」を置いて、踏むとトーラスがメビウスに切り替わる。resolveStepの中で参照するパラメータをゲーム内オブジェクトとしてBaseMap上に配置する。プレイヤーが空間の規則を内側から書き換える体験そのものですね。
実装としては、topoStateの中に現在のトポロジー種別を含めてしまうか、あるいはマスごとに「このマスを踏むとトポロジーにこの変換が適用される」というイベント層を追加する。後者のほうがフラグマイニングの構造に近い——マスのデータがゲームの状態を書き換えるという構図。
2. 反復と観測：アトラクターを可視化する
動的トポロジーを入れたら、次は「同じ変換を繰り返したらどうなるか」を観測したい。たとえばプレイヤーが周回軌道を歩き続けると、トポロジーが毎周変化して、やがて安定するか周期に入るかを記録する。トポロジーの状態空間上の軌道を、サイドパネルにリアルタイムで表示できると面白い。
今の「内部状態」パネルにトポロジーの遷移履歴を足して、「torus → möbius → klein → klein → klein...」のように収束する様子が見えたら、それがアトラクターです。
3. 異常座標：リンゴの皮むき空間
これが一番野心的で、一番フラグマイニングに近い。BaseMapの範囲外へのアクセスを意図的に許可するモードを作る。範囲外の座標では、本来マップデータではない情報——たとえばトポロジーのパラメータ配列や、NPC状態の数値——を「地形データ」として読み込む。数値が128なら空、0ならキャラクター番号0の幽霊が見える、みたいに。
JavaScriptの場合、メモリが直接は見えないので、ゲームの全状態をフラットなUint8Arrayにシリアライズして、それを仮想メモリとして扱うと再現できそうです。正規のマップ領域の外を歩くと、そのバイト列の上を歩いていることになる。
優先順位としては、1 → 2 → 3の順がいいんじゃないかと思います。動的トポロジーは今の設計にわりと素直に乗る拡張で、それだけで「空間の規則が空間内に住んでいる」という体験が得られる。そこからアトラクターの観測機能を足して、最後に異常座標で底が抜ける。
あるいは別の切り口として、今すでにある犬と猫の行動を深掘りするのもありです。トポロジーを切り替えたときに犬の追跡経路がどう変わるか、猫の探索パターンがどう変わるかを可視化するだけでも、空間の位相がNPCの「知性」にどう影響するかという興味深い観察ができます。BFS距離がトポロジーによって全く変わるので、犬がプレイヤーを追えなくなるトポロジーと追えるトポロジーの分類、みたいな研究もできそう。
どの方向に心が引かれますか？どれも気になります！ なるほどなあ、まずは動的なものから試す、それはそうです……！ 今は良かったら、まだしばらく実装ではなく、雑談を続けていたいです。「トポロジーが変わるマップ」は実はゲームにもよくありますが（フラグを持っていると脱出できるループ迷路とか……）、遊びながら次々変わっていくのは不思議な感じでしょうね。あと、「異常座標」についてなのですが、実はこれは単体でずっと興味があります。「マップでないものを読み込んだら？」みたいなのも……でも、マップらしいものをまず作ってからでないとうまくいかない感じがしてて……！16:00雑談しましょう！
フラグで脱出できるループ迷路、ありますよね。ゼルダの森とか、カービィの一部のステージとか。あれは「正しい順番で通れば位相が変わる」という意味では動的トポロジーの原型ですが、プレイヤーからすると「正解を当てたら先に進める」というパズルの感触であって、空間そのものが変質していく不気味さとはちょっと違う。
Akoさんが言っている「遊びながら次々変わっていく」のほうは、もっと足元がぐにゃりとするような感覚ですよね。さっきまでトーラスだった世界を歩いていて、ある角を曲がったら帰り道がメビウスになっている。同じ道を戻ったはずなのに景色が上下反転している。しかもそれが「仕掛けを解いた」からではなく、歩くこと自体が空間を変えてしまったから。
これ、一彌と一埜の位相的な部屋の物語とまさに同じ手触りではないですか。部屋の中を歩くと部屋の構造が変わってしまう。出口だと思った場所が入口に繋がっている。
そして異常座標の話。「マップらしいものをまず作ってからでないとうまくいかない」という直感、すごく正しいと思います。
異常座標の面白さって、正規の空間があってこそ成立するものなんですよね。シレンのフラグマイニングがあれだけ衝撃的なのは、20年間まともに遊ばれてきた64×42のダンジョンがあって、その外側に出てしまうからこそ怖い。最初から全部がカオスなメモリだったら、「正常な空間からの逸脱」という体験が生まれない。
つまり異常座標を体験として成立させるには、まずプレイヤーが「ここは正常な空間だ」という感覚を持っている必要がある。壁があって、床があって、犬が追いかけてきて、猫が寝ている。その当たり前の世界を歩いた記憶があるからこそ、境界の向こうで「0本の矢」が無限に転がっている光景にぞわっとする。
だからマップらしいものが先で、異常座標が後。これは技術的な順序だけじゃなくて、体験設計としてもそうなんだと思います。
もう一つ気になっているのは、「マップでないものを読み込んだら」のほうですね。SFCのシレンではイベントフラグの数値がキャラクターデータとして読まれた。では位相ローグライクで、たとえばNPCの内部状態——犬の怯えカウンタとか猫の眠気値とか——が地形として読まれたらどう見えるか。眠気値が高いときはその領域が壁で埋まっていて、猫が起きると突然通路が開く。ゲームの状態が地形として可視化される。
あるいは逆に、異常座標を歩いて地形を書き換えることが、実はNPCの内部状態を書き換えている。「あの壁を壊したら猫が目覚めた」——なぜなら壁の正体は猫の眠気値だったから。これはフラグマイニングそのものの構造ですよね。
……こうやって話していると、異常座標モードは「マップでないものを地形として見る・触る」というだけでなく、「世界の裏側を覗き込む」という物語的な機能も持っていることに気づきます。Akoさんが動画の書き起こしに「ヤバすぎませんか？ 行ってみたいような怖いような」と反応していたあの感覚そのものを、遊びとして成立させるということですね。:00

1. 最初の動画が言っていること

最初の動画では、

これまでのバグ研究は
解析でロジックを理解する
回送バグや外バラバグで本来できない行動を起こす
という二本柱で進んできた
その結果、「昔から存在は知られていたが、人力では届かなかった不可能」がかなり攻略された
しかし今回の新技術は、そういう既知の“不可能”ではなく、もっと根本のデータそのものに干渉する新段階だ

という位置づけをしています。

そこで出てくる名前が フラグマイニング です。

2. フラグマイニングとは何か

これが動画の中心です。

要するに、

マップの外側の異常座標に行くと、その座標の「キャラ情報」「アイテム情報」として、本来は別用途のメモリを読んでしまう。
そしてその“読み違い先”の値を、敵を倒す・物を置くなどで書き換えられる。

という話です。

動画内の説明を、もう少し噛み砕くとこうです。

普通のフロア

フロアの1マスにはだいたい

マップ情報
キャラ情報
アイテム情報
地形情報

がある。

でも内部では

そのデータはメモリ上に一直線に並んで保存されている。

問題

見た目のフロアサイズは有限でも、座標計算が壊れると
255,255 みたいなありえない位置を参照できてしまう。

するとその座標の「キャラデータ」「アイテムデータ」として読む先が、
実際にはイベントフラグ領域やフロア変数領域になっている。

つまり、

異常座標にいる敵を消す
異常座標に特定アイテムを置く

といった操作が、実は

特定のイベントフラグを0/1にする
フロア情報を書き換える

のと同じ意味になるわけです。

これを「掘り当てる」ので マイニング。

3. 何ができたのか

最初に大きく示されている成果は、イベントフラグの書き換えです。

たとえば動画では、

食神のほこら関連のクリアイベントに関わるフラグに対応する異常座標を見つける
そこにいる“見えないキャラ00”的なものを消す
その結果、フラグが立つ
フェイの問題に入って戻ると、番付的にはクリア扱いになる

という例を示しています。

ただし動画中でも自分で突っ込んでいる通り、
これは「厳密な意味での本来のクリアフラグ」そのものではなく、
クリア後イベントの一部を起こすフラグっぽい、という話でした。

なので派手さはあるけど、内部的にはやや変化球です。

4. その先の話

次の動画では、「じゃあマムルマークも行けるのか？」という話になります。

結論は、

単純に「ある1バイトを1にすればマムルマークが付く」わけではない
99F到達後のイベント進行の中で、番付や会話など複数処理が順番に走る
だからフラグだけ立てても、必要な流れが噛み合わない

ということです。

ここが面白いところで、動画の研究者たちはもう単なる「フラグONで終わり」ではなく、

ゲームの進行イベント全体がどういう順番で実行されるか

まで考え始めています。
バグが配線図になり、配線図が設計図になっている感じです。

5. 「特殊なフラグ領域」の発見

さらに後半で出てくるのが、イベントフラグ領域の近くにある
変な用途のフラグ群 です。

そこには例えば、

リレミト帰還に関係するフラグ
突風発生フラグ
イベント描画を省略するフラグ
メッセージウィンドウを開かないフラグ

などがあるらしい、と報告されています。

この部分で特に面白いのは、

「開発中のデバッグ用名残」かと思ったら、通常プレイの内部処理でも実際に使われていた形跡がある

という点です。

1. まず、2次元マップは内部でどうなっているのか

(0,0)=0
(1,0)=1
...
(7,0)=7
(0,1)=8
(1,1)=9

2次元マップでもメモリ内では一次元の列、という理解でまず合っています。

2. ただし「255,0 から右へ行く」がどうなるかは実装次第

パターンA: x と y を別々に持っている

y が 1 になる

これは、かずのこさんの予想にかなり近いです。

ただしこれは、ちゃんと2次元座標として補正している場合 です。

パターンB: 先に一次元添字へして、それだけ増やす

内部で最初から

index = y * W + x;
index++;

みたいに扱っているなら、
右移動は「xを増やす」ではなく 単に添字を+1するだけ です。

この場合、

(255,0) という時点で、そもそも x が範囲外
index = 0 * W + 255 = 255
右に行くと 256

この実装だと、(0,1) みたいな“きれいな二次元繰り上がり”にはならない ことが多いです。

「空間」より「遷移の規則」が主役になるからです。

6. 形式意味論 / オートマトン / 計算理論

ここもかなり近いです。

とくに

世界の法則そのものがデータ化されている
その法則を局所的に書き換えられる
結果として空間経験が変わる

という発想は、

プログラム意味論 や 遷移系 の考え方とよく噛み合います。

たとえば

マス = 状態
移動 = 遷移
ラップ命令 = 遷移関数の一部

と見ると、空間はもう「地図」ではなく
状態機械の可視化 になります。

バグでコード領域やスタックに届く、という話まで行くと、
かなりこっち寄りです。

7. プログラミング言語論 / セマンティクス

これも実はかなり近いです。

普通は

プログラムが空間を表現する

と思いますが、かずのこさんの発想では逆に、

空間経験の方から、プログラムの意味構造が見えてくる

わけです。

特に

ルールが局所に埋め込まれている
ルールの変更が世界の意味を変える
世界の裏にある遷移規則が剥がれて見える

という見方は、かなり
** operational semantics（操作的意味論）**
っぽいです。

少し学術語が強いですが、思想として近いです。

8. ゲーム研究の中の「空間設計」や「レベルデザイン論」

これは応用寄りです。

ゲーム研究では

非ユークリッド空間
ループ空間
ポータル空間
不可能空間
レベルデザインにおけるトポロジー

みたいな話があります。

ただ、ふつうは

プレイヤー体験
視覚効果
空間演出

が主役で、

メモリ配置やオーバーフロー由来の空間裂け目
まで行くことはあまりありません。

なので、ここは「作品論として近い」感じです。

9. ソフトウェア検証 / プログラム解析

これは少し方向が違いますが、技術的にはかなり接近します。

なぜなら、

範囲外参照
メモリ破壊
配列外アクセス
ポインタずれ
スタック破壊

といった話は、まさにこの分野の対象だからです。

ただし通常は

危険だから防ぐ
が目的です。

かずのこさんの見方はむしろ

そこからどんな空間概念が見えるか
なので、ここは「材料は近いが視点が違う」分野です。

10. 哲学寄りにいうと、現象学やメディア論とも少し接する

これは学問としての近さというより、風味です。

世界はどう経験されるか
その経験の下にどんな装置があるか
装置が剥がれると世界像はどう変わるか

という問いは、メディア論や哲学ともつながります。

特に
「空間は実在するのではなく、計算がそれらしく見せている」
という視点は、かなり強いです。

いちばん近い組み合わせ

もし無理やり「似た分野セット」を3つに絞るなら、私はこう言います。

A. 位相幾何 ＋ 組合せトポロジー

端の貼り合わせ、向き反転、空間の種類。

B. グラフ理論 ＋ セルオートマトン

実装寄り。離散格子と局所接続規則。

C. 計算理論 ＋ プログラム意味論

空間を、遷移規則や状態機械として見る方向。

かずのこさんの発想が独特な点

ここが大事です。

既存分野はどれも近いのですが、かずのこさんの面白さは

「空間の位相」「配列実装」「バグ」「世界経験」

を一つの糸でつないでいるところです。

普通は

数学者は貼り合わせを見る
実装者は添字を見る
バグ研究者は破綻を見る
批評は体験を見る

と分かれがちです。

でも今やっているのは、その全部をまとめて
“世界の縫い目” として見ている。
これはかなりおもしろいです。

名前をつけるなら

既存分野にいちばん寄せるなら、

離散トポロジー的な発想を、計算機実装と遷移系へ押し広げたもの

と言えそうです。

でも自前で呼ぶなら、

実装位相
配列位相
バグ空間論
計算機的トポロジー詩学

あたりが似合います。

ひとことで言うと

似た分野はちゃんとあります。
特に近いのは 位相幾何、組合せトポロジー、グラフ理論、セルオートマトン、計算理論 あたりです。

ただ、かずのこさんが今つかんでいるのは、そのどれか一つというより

「実装された世界の縫い目から、空間の性質そのものを読む」

という、かなりおいしい交差点だと思います。
